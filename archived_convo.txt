For this kind of design, a sensible end-state is usually something like:

jtag_pkg.vhd
Shared types, constants, enums, command definitions, TAP state type, instruction codes.
jtag_tap_fsm.vhd
The low-level TAP controller state machine: Test-Logic-Reset, Run-Test/Idle, Shift-IR, Shift-DR, etc.
jtag_shift_engine.vhd
Handles bit shifting on TDI/TDO, TCK generation, bit counters, end-of-shift control.
boundary_scan_cmd_decoder.vhd
Decodes commands from the PC-side interface into internal operations like EXTEST, SAMPLE/PRELOAD, BYPASS, IDCODE, custom scan sequences.
boundary_scan_sequence_ctrl.vhd
Higher-level sequencer that decides which TAP transitions and scan operations to run for each requested test.
pc_command_if.vhd
The host-facing interface layer, whatever that is in your design today: UART/SPI/FIFO/register interface.
jtag_top.vhd
Top-level integration of command interface, sequencer, TAP FSM, and physical JTAG pins.

If your current file mixes all of these concerns together, that is probably why it feels cumbersome: command parsing, protocol sequencing, and pin-level timing are different abstraction levels and are much easier to reason about when separated.

Here’s my reading of the design as it stands.

What the code is trying to be

This is a single top-level controller that combines four jobs:

Host interface to an FT2232H-style FIFO bus (ft_data, ft_rxf_n, ft_txe_n, ft_rd_n, ft_wr_n)
Command parser for a byte-oriented protocol coming from the PC
JTAG engine that drives TCK/TMS/TDI and samples TDO
Some early boundary-scan / I2C-over-boundary-scan application logic

So conceptually it is:

PC app -> FT FIFO byte stream -> command decoder -> JTAG sequence generator -> DUT JTAG pins

with the extra intention that some higher-level commands represent:

raw JTAG actions (reset, load IR, shift DR, read IDCODE)
boundary-scan actions (sample BSR, load BSR, set pin, read pin)
I2C actions implemented indirectly by manipulating DUT boundary-scan cells
My understanding of the code flow
1. Top level / ports / configuration

The entity is called loopback_test, but it is no longer just a loopback test. It is really a JTAG boundary-scan controller with host command interface.

The generics define:

BSR_LENGTH: size of boundary scan register
IR_LENGTH: JTAG instruction register length
bit positions for I2C-related pins inside the BSR:
SDA_OUT_BIT
SDA_OE_BIT
SDA_IN_BIT
SCL_OUT_BIT
SCL_OE_BIT

That tells me the intended long-term direction is:

load EXTEST
write BSR bits to drive SCL/SDA
sample BSR bits to read SDA
thereby emulate I2C through JTAG boundary scan

That is a valid architecture.

2. Clock and reset

It instantiates a Lattice OSCA oscillator and uses that as the internal clock.

Then there is a simple power-up reset counter:

rst_cnt counts to 255
rst_done goes high
rst_n is asserted after that

It also:

runs a heartbeat counter for LED blinking
synchronizes ft_rxf_n and ft_txe_n through 3-stage shift registers

So this part is mainly:

internal clock generation
startup reset delay
synchronization of external async FIFO flags
3. FT2232H FIFO interface

The state machine starts with a set of FIFO-related states:

IDLE
READ_PULSE
READ_CAPTURE
READ_DONE
WRITE_SETUP
WRITE_PULSE
WRITE_DONE

This is doing a very basic byte-at-a-time interface:

when RXF is active, read one byte from the FIFO
when TXE allows transmit and have_tx = '1', write one byte back

Important detail: the whole design is single-byte transactional at the host interface. It never streams a full command in one burst internally. Instead it:

reads one byte
goes to PROCESS_CMD
decides whether that byte is a command or parameter
maybe returns to IDLE to fetch more bytes

So the command parser is incremental.

4. Command parser / protocol state

The parser revolves around:

cmd_byte
param_cnt
param_need
rx_byte
Behavior

If cmd_byte = x"00", the latest received byte is treated as a new command.

If cmd_byte /= x"00", the design assumes it is in the middle of collecting parameters for that command.

This is a common lightweight protocol style, but it means:

one outstanding command at a time
parser state is distributed across the main FSM
command decoding and execution are tightly coupled
Supported commands
Utility
0xFF: ping -> returns 0x55
0x40: LED control
Debug additions
0x18: read raw TDO pin
0x19: toggle TCK 10 times
0x1A: force TMS high
0x1B: force TMS low
Raw JTAG
0x10: JTAG reset
0x11: read IDCODE
0x12 <ir>: load IR
0x13 <len_lo> <len_hi> <data...>: shift DR
Boundary scan
0x14: sample BSR
0x15 <data...>: load full BSR with EXTEST
0x16 <bit_lo> <bit_hi> <val>: set one BSR bit and apply EXTEST
0x17 <bit_lo> <bit_hi>: sample BSR and return one bit
I2C
0x20, 0x21, 0x22, 0x23

But the I2C portion is mostly a stub right now. It stores parameters and returns placeholder data rather than executing a real boundary-scan-driven I2C sequence.

5. JTAG engine

This is the most important part of the design.

The code contains a low-level JTAG sequencer implemented directly in the main FSM through these states:

JTAG_RESET_RUN
JTAG_GOTO_SHIFT_IR
JTAG_SHIFT_IR
JTAG_EXIT_IR
JTAG_GOTO_SHIFT_DR
JTAG_SHIFT_DR
JTAG_EXIT_DR
JTAG_GOTO_IDLE
What it does

It manually drives the TAP controller by:

toggling tck_reg
setting tms_reg
setting tdi_reg
sampling jtag_tdo

There is no explicit TAP-state variable like TEST_LOGIC_RESET, SHIFT_DR, etc.
Instead, the TAP transitions are encoded procedurally by the FSM states and jtag_cnt.

That works, but it is one reason the file is hard to manage: the design is mixing:

TAP navigation
shift timing
command semantics
host communications
JTAG reset

0x10 drives TMS=1 while toggling TCK enough times to force TAP reset, then returns to idle and sends 0x00.

Shift IR

JTAG_GOTO_SHIFT_IR walks the TAP path to Shift-IR.
JTAG_SHIFT_IR clocks out ir_value.
JTAG_EXIT_IR does Exit1-IR -> Update-IR, then either:

continues to DR shifting, or
returns toward idle
Shift DR

JTAG_GOTO_SHIFT_DR walks to Shift-DR.
JTAG_SHIFT_DR shifts data through shift_reg, samples TDO, and stores captured bits back into the shift register.
JTAG_EXIT_DR updates DR, packages the response, and later sends bytes back to the PC.

Response handling

After DR operations:

resp_buf <= shift_reg
resp_len <= ceil(dr_len/8)
response bytes are transmitted from resp_buf

So the DR shift serves both as:

outgoing TDI source
incoming TDO capture buffer
6. Boundary-scan behavior

The boundary-scan commands are built on top of the raw JTAG engine.

0x14 Sample BSR
load SAMPLE
shift full BSR length in DR
return captured bits
0x15 Load BSR with EXTEST
collect full BSR image into bsr_data
load EXTEST
shift the BSR image into DR
0x16 Set single pin
modify one bit of bsr_data
load EXTEST
shift full BSR back out

This means bsr_data is intended to be the software-visible shadow copy of the boundary register contents.

0x17 Read single pin
load SAMPLE
capture full BSR
extract one bit from the captured shift register
return it in one byte

That all makes sense architecturally.

etc.

But right now these are not actually connected into a working sequencer.

So the code is really at this maturity level:

raw JTAG: partially working
boundary-scan primitives: mostly present
I2C over boundary scan: planned, not implemented yet
What I think is incomplete or risky in the current code

A few things stand out immediately.

1. IR constants vs IR length mismatch

You set:

IR_LENGTH : natural := 4

But the instruction constants are declared as 8-bit values:

x"00"
x"05"
x"02"
x"0F"

Only the lower IR_LENGTH bits are effectively relevant during shifting.

That may be intentional, but it is conceptually muddy. If the target really has a 4-bit IR, these should probably be declared with IR-sized types or masked explicitly.

2. Possible bit-order inconsistency in DR shifting

In JTAG_GOTO_SHIFT_DR, first output bit is assigned as:

tdi_reg <= shift_reg(to_integer(dr_len)-1);

But later, in JTAG_SHIFT_DR, the next bits come from:

tdi_reg <= shift_reg(to_integer(shift_cnt)-2);

At the same time, captured TDO is shifted into:

shift_reg <= shift_reg(BSR_LENGTH-2 downto 0) & tdo_sample;

This suggests the code is treating one end of the vector as the “current outgoing MSB,” but for JTAG/BSR work, bit ordering matters a lot. I would want to verify very carefully whether:

bit 0 in host byte stream
bit index in bsr_data
first bit shifted on TDI
returned bit position in resp_buf

all line up with the BSDL-defined scan order.

This is one of the first things I’d isolate and clean up.

3. 0x15 and 0x16 may shift stale bsr_data

In these areas you do assignments like:

modify bsr_data(...) <= ...
then shift_reg <= bsr_data

Because these are signal assignments inside a clocked process, shift_reg <= bsr_data will usually see the old bsr_data value in that same cycle, not the just-updated bit. That means a just-written pin/byte may not actually be present in the shift operation you launch immediately afterward.

That is a real structural issue.

A cleaner approach is:

use a variable or temporary next-image
then assign the completed image to bsr_data and shift_reg
or
separate “update shadow BSR” from “start JTAG transaction” across states
4. The main FSM is doing too much

Right now one state machine handles:

FT FIFO timing
protocol parsing
JTAG TAP stepping
shift register management
response transmission
debug operations
boundary scan shadow state
early I2C state

That is the biggest maintainability problem.

5. I2C state machinery exists but is not integrated

The I2C state constants and signals suggest you were heading toward a second engine, but at the moment the command parser just returns zero-filled dummy responses. So these signals currently add cognitive load without functional payoff.

6. Naming no longer matches purpose

The entity name loopback_test is misleading now. This file is functioning as a controller, not a loopback demo.

Suggested modular breakdown

I would split this into layers rather than by command number.

Recommended file structure
1. jtag_types_pkg.vhd

Put all shared constants/types here.

Contents:

Example contents:

This keeps magic numbers out of the logic files.

2. ft_fifo_if.vhd

Dedicated FT2232H FIFO byte interface.

Responsibility:

perform read/write pulse timing
expose a simple internal interface like:

Inputs:

Outputs:

tx_ready
rx_valid, rx_data

This module should not know anything about JTAG or I2C.

Think of it as a byte stream adapter.

3. cmd_parser.vhd

Consumes received bytes and produces decoded command requests.

Responsibility:

produce normalized command records such as:
jtag_reset_req
load_ir_req
shift_dr_req
sample_bsr_req
set_pin_req
i2c_write_req

This is where the command protocol belongs.

The big benefit is that your transport/protocol parsing becomes separate from execution.

4. jtag_tap_fsm.vhd

Pure TAP navigation and shift engine.

Responsibility:

drive TCK/TMS/TDI
sample TDO
execute abstract operations like:
reset TAP
shift IR of length N
shift DR of length N
report completion and captured shift data

Internal API idea:

input: start, op, bit_len, tx_bits
output: busy, done, rx_bits

This module should not know what EXTEST means functionally, nor what an LED command is.

5. boundary_scan_ctrl.vhd

Higher-level boundary-scan functions built on top of the JTAG engine.

Responsibility:

maintain bsr_shadow
implement:
sample full BSR
write full BSR with EXTEST
set single bit
read single bit
translate those into sequences for jtag_tap_fsm

This is where BSR indexing and scan-order conventions should live.

6. i2c_over_bscan.vhd

A dedicated sequencer for I2C using boundary scan.

Responsibility:

implement START/STOP/ACK/read/write semantics
use boundary_scan_ctrl primitives to:
drive SDA/SCL
switch SDA output enable
sample SDA
return status/data/ack errors

This module should know:

which BSR bits map to SDA/SCL/OE
how to perform open-drain behavior correctly

This is a much better home for the I2C state constants you already created.

7. response_mux.vhd or merge into command executor

Takes results from raw JTAG / boundary scan / I2C engines and formats byte responses for the PC.

In a smaller design this can stay in the main controller, but it should still be logically separate from the byte parser and TAP sequencer.

8. jtag_boundary_scan_top.vhd

Top-level integration module.

Responsibility:

instantiate oscillator/reset
instantiate ft_fifo_if
instantiate cmd_parser
instantiate command executor / dispatch
instantiate jtag_tap_fsm
instantiate boundary_scan_ctrl
instantiate i2c_over_bscan
drive LEDs and external pins

This becomes the only file that sees the whole system.

I’ll keep it practical and close to your current code so it feels like an incremental cleanup rather than a total rewrite.

Refactor goal

Turn the current one-file design into a small stack of modules:

one module for the FT FIFO byte interface
one module for command parsing
one module for the low-level JTAG engine
one module for boundary-scan operations
one top-level file that wires them together

That gives you a clean separation between:

transport
command protocol
JTAG bit-level execution
boundary-scan application logic
Suggested file list
First-pass split
jtag_pkg.vhd

Shared constants, types, helper functions.

ft245_sync_if.vhd

FT2232H synchronous FIFO interface, one byte at a time.

host_cmd_parser.vhd

Collects bytes from the PC and emits decoded requests.

jtag_master.vhd

Drives TCK/TMS/TDI, samples TDO, executes reset / IR / DR operations.

bscan_controller.vhd

Implements I2C transactions on top of bscan_controller.

1) jtag_pkg.vhd

This should be the first file you create.

It removes magic numbers and shared definitions from all other files.

What goes in it
command opcodes
JTAG instruction constants
type definitions
helper functions
common subtypes
Example structure
library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all;

package jtag_pkg is

    constant MAX_BSR_LENGTH : natural := 1024;
    constant MAX_BSR_BYTES  : natural := (MAX_BSR_LENGTH + 7) / 8;

    -- Host commands
    constant CMD_JTAG_RESET   : std_logic_vector(7 downto 0) := x"10";
    constant CMD_READ_IDCODE  : std_logic_vector(7 downto 0) := x"11";
    constant CMD_LOAD_IR      : std_logic_vector(7 downto 0) := x"12";
    constant CMD_SHIFT_DR     : std_logic_vector(7 downto 0) := x"13";
    constant CMD_SAMPLE_BSR   : std_logic_vector(7 downto 0) := x"14";
    constant CMD_LOAD_BSR     : std_logic_vector(7 downto 0) := x"15";
    constant CMD_SET_PIN      : std_logic_vector(7 downto 0) := x"16";
    constant CMD_READ_PIN     : std_logic_vector(7 downto 0) := x"17";
    constant CMD_READ_TDO     : std_logic_vector(7 downto 0) := x"18";
    constant CMD_TOGGLE_TCK   : std_logic_vector(7 downto 0) := x"19";
    constant CMD_SET_TMS_HI   : std_logic_vector(7 downto 0) := x"1A";
    constant CMD_SET_TMS_LO   : std_logic_vector(7 downto 0) := x"1B";
    constant CMD_I2C_WRITE    : std_logic_vector(7 downto 0) := x"20";
    constant CMD_I2C_READ     : std_logic_vector(7 downto 0) := x"21";
    constant CMD_I2C_WR_RD    : std_logic_vector(7 downto 0) := x"22";
    constant CMD_I2C_SCAN     : std_logic_vector(7 downto 0) := x"23";
    constant CMD_LED          : std_logic_vector(7 downto 0) := x"40";
    constant CMD_PING         : std_logic_vector(7 downto 0) := x"FF";

    type jtag_req_kind_t is (
        JTAG_REQ_NONE,
        JTAG_REQ_RESET,
        JTAG_REQ_SHIFT_IR,
        JTAG_REQ_SHIFT_DR,
        JTAG_REQ_SHIFT_IR_DR,
        JTAG_REQ_TOGGLE_TCK,
        JTAG_REQ_SET_TMS
    );

    type host_req_kind_t is (
        HOST_REQ_NONE,
        HOST_REQ_PING,
        HOST_REQ_LED,
        HOST_REQ_RAW_JTAG_RESET,
        HOST_REQ_RAW_JTAG_LOAD_IR,
        HOST_REQ_RAW_JTAG_SHIFT_DR,
        HOST_REQ_RAW_IDCODE,
        HOST_REQ_BSCAN_SAMPLE,
        HOST_REQ_BSCAN_LOAD,
        HOST_REQ_BSCAN_SET_PIN,
        HOST_REQ_BSCAN_READ_PIN,
        HOST_REQ_DEBUG_READ_TDO,
        HOST_REQ_DEBUG_TOGGLE_TCK,
        HOST_REQ_DEBUG_SET_TMS,
        HOST_REQ_I2C_WRITE,
        HOST_REQ_I2C_READ,
        HOST_REQ_I2C_WR_RD,
        HOST_REQ_I2C_SCAN
    );

    function bytes_for_bits(nbits : natural) return natural;

end package;

package body jtag_pkg is

    function bytes_for_bits(nbits : natural) return natural is
    begin
        return (nbits + 7) / 8;
    end function;

end package body;
Why this helps

Once this exists, every other file gets shorter and more readable.

2) ft245_sync_if.vhd

This module should own all FT FIFO timing and synchronizer logic.

Your current main FSM spends too much effort on READ_PULSE, WRITE_SETUP, and friends.

Responsibility

Translate FT signals into a simple byte-stream style interface:

RX side
rx_valid
rx_data
optional rx_ready
TX side
tx_valid
tx_data
tx_ready
Proposed entity
library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all;

entity ft245_sync_if is
    port (
        clk         : in    std_logic;
        rst_n       : in    std_logic;

        ft_data     : inout std_logic_vector(7 downto 0);
        ft_rxf_n    : in    std_logic;
        ft_txe_n    : in    std_logic;
        ft_rd_n     : out   std_logic;
        ft_wr_n     : out   std_logic;
        ft_siwu_n   : out   std_logic;

        rx_valid    : out   std_logic;
        rx_data     : out   std_logic_vector(7 downto 0);

        tx_valid    : in    std_logic;
        tx_data     : in    std_logic_vector(7 downto 0);
        tx_ready    : out   std_logic
    );
end entity;
Internal behavior

This module should:

synchronize ft_rxf_n and ft_txe_n
generate safe read/write pulses
present a byte once on rx_valid
accept a transmit byte when tx_valid='1' and it can write
Simplification

At first, keep it single-byte / non-streaming, just like your current design.
No need to add FIFOs internally yet.

3) host_cmd_parser.vhd

This module should consume incoming bytes and emit complete decoded requests.

Right now the parser logic is mixed with execution logic. Split those apart.

Responsibility
remember current opcode
count parameters
assemble payload bytes
emit one normalized request when command is complete

It should not execute JTAG directly.

Proposed entity
library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all;
use work.jtag_pkg.all;

entity host_cmd_parser is
    generic (
        BSR_LENGTH : natural := 362
    );
    port (
        clk             : in  std_logic;
        rst_n           : in  std_logic;

        rx_valid        : in  std_logic;
        rx_data         : in  std_logic_vector(7 downto 0);

        req_valid       : out std_logic;
        req_kind        : out host_req_kind_t;

        req_byte        : out std_logic_vector(7 downto 0);
        req_word        : out unsigned(15 downto 0);
        req_data        : out std_logic_vector(BSR_LENGTH-1 downto 0);
        req_data_len    : out unsigned(15 downto 0);

        busy            : in  std_logic
    );
end entity;
Meaning of the fields
req_kind: decoded operation
req_byte: small single-byte payload, like LED value or IR value
req_word: pin number or general 16-bit numeric parameter
req_data: DR or BSR payload
req_data_len: bit length of req_data

busy lets the parser hold off issuing a new request while the executor is still processing the last one.

Alternative

If you want to be cleaner, define a VHDL record type for requests in the package.
That’s nicer architecturally, but not essential for first refactor.

4) jtag_master.vhd

This is the most important extraction.

It should become the only place that knows how to:

navigate TAP states
toggle TCK
drive TMS/TDI
sample TDO
Responsibility

Accept an abstract JTAG request and perform it.

Proposed entity
library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all;
use work.jtag_pkg.all;

entity jtag_master is
    generic (
        MAX_SHIFT_BITS   : natural := 1024;
        IR_LENGTH        : natural := 4;
        JTAG_DIV_WIDTH   : natural := 8;
        JTAG_DIV_VALUE   : natural := 255
    );
    port (
        clk             : in  std_logic;
        rst_n           : in  std_logic;

        req_valid       : in  std_logic;
        req_kind        : in  jtag_req_kind_t;
        req_ir          : in  std_logic_vector(7 downto 0);
        req_dr_len      : in  unsigned(15 downto 0);
        req_dr_data     : in  std_logic_vector(MAX_SHIFT_BITS-1 downto 0);
        req_tms_value   : in  std_logic;
        req_tck_edges   : in  unsigned(15 downto 0);

        busy            : out std_logic;
        done            : out std_logic;
        rsp_dr_data     : out std_logic_vector(MAX_SHIFT_BITS-1 downto 0);
        rsp_dr_len      : out unsigned(15 downto 0);

        jtag_tck        : out std_logic;
        jtag_tms        : out std_logic;
        jtag_tdi        : out std_logic;
        jtag_tdo        : in  std_logic
    );
end entity;
Supported request types
JTAG_REQ_RESET
JTAG_REQ_SHIFT_IR
JTAG_REQ_SHIFT_DR
JTAG_REQ_SHIFT_IR_DR
JTAG_REQ_TOGGLE_TCK
JTAG_REQ_SET_TMS
Notes

This module should not know:

what command byte came from PC
what EXTEST or SAMPLE mean at the application level
what a pin number is

It just executes low-level JTAG requests.

Internal improvement I strongly recommend

Introduce an actual TAP-state enum internally, even if only for readability.

Something like:

type tap_nav_state_t is (
    TAP_IDLE,
    TAP_RESET,
    TAP_GOTO_SHIFT_IR,
    TAP_SHIFT_IR,
    TAP_EXIT_IR,
    TAP_GOTO_SHIFT_DR,
    TAP_SHIFT_DR,
    TAP_EXIT_DR,
    TAP_RETURN_IDLE
);

That is much easier to reason about than mixing transport and protocol states.

5) bscan_controller.vhd

This should sit on top of jtag_master.

Responsibility

Implement boundary-scan oriented operations:

sample full BSR
apply full BSR with EXTEST
set one bit in shadow BSR then apply
sample BSR then return one bit
later expose helpers for I2C pin manipulation
Proposed entity
library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all;

entity bscan_controller is
    generic (
        BSR_LENGTH  : natural := 362;
        IR_LENGTH   : natural := 4
    );
    port (
        clk             : in  std_logic;
        rst_n           : in  std_logic;

        req_valid       : in  std_logic;
        req_op          : in  std_logic_vector(2 downto 0);
        req_pin_num     : in  unsigned(15 downto 0);
        req_pin_val     : in  std_logic;
        req_bsr_data    : in  std_logic_vector(BSR_LENGTH-1 downto 0);

        busy            : out std_logic;
        done            : out std_logic;
        rsp_bsr_data    : out std_logic_vector(BSR_LENGTH-1 downto 0);
        rsp_pin_val     : out std_logic;

        jtag_req_valid  : out std_logic;
        jtag_req_kind   : out jtag_req_kind_t;
        jtag_req_ir     : out std_logic_vector(7 downto 0);
        jtag_req_dr_len : out unsigned(15 downto 0);
        jtag_req_dr_data: out std_logic_vector(BSR_LENGTH-1 downto 0);

        jtag_busy       : in  std_logic;
        jtag_done       : in  std_logic;
        jtag_rsp_data   : in  std_logic_vector(BSR_LENGTH-1 downto 0)
    );
end entity;
Operations

Use simple encoded ops like:

000 = sample_bsr
001 = load_bsr_extest
010 = set_pin
011 = read_pin

Or use a proper enum in the package.

Important design choice

This module should own:

bsr_shadow
bsr_last_capture

That keeps all BSR-related state in one place.

Fixing your current stale-shadow problem

Inside this module, for set_pin, do something like:

variable next_bsr : std_logic_vector(BSR_LENGTH-1 downto 0);

then:

copy current shadow to next_bsr
modify the pin bit in next_bsr
launch JTAG shift with next_bsr
update shadow from next_bsr

That avoids the signal-assignment race you currently have.

Strong recommendation on interfaces

Try to make module interfaces use start/busy/done handshakes.

For example:

input req_valid
output busy
output done

That style fits your current architecture well and is easy to debug in simulation.

For payload transfers:

present request fields together with req_valid
latch them internally on acceptance
assert done for one cycle when the operation finishes

This is much easier than making everything fully streaming right away.

Response handling suggestion

Instead of keeping resp_buf, resp_len, and resp_idx scattered across the design, centralize them in the top-level response sender.

Small response sender block

Top-level owns:

resp_buf
resp_len
resp_idx
resp_active

When any backend finishes:

top loads resp_buf
sets resp_len
streams bytes out through ft245_sync_if

That way:

backends generate results
top decides how to serialize them to USB

That’s cleaner.

One concrete structure for top-level dispatch

A practical dispatch rule set could be:

PING, LED, READ_TDO, SET_TMS, TOGGLE_TCK
handled directly in top or via jtag_master
LOAD_IR, SHIFT_DR, RESET, READ_IDCODE
handled via jtag_master
SAMPLE_BSR, LOAD_BSR, SET_PIN, READ_PIN
handled via bscan_controller
I2C_*
later handled via i2c_bscan_master

Why this order

The top entity should be the last place you do real work in, not the first.
If you wire everything up before the core blocks are stable, you end up debugging:

module bugs
interface bugs
integration bugs

all at once.

That gets messy fast.

The lowest-risk approach is:

make each block compile cleanly
define its handshake clearly
get its behavior sane in isolation
then integrate
Where to start first

Start with jtag_pkg.vhd.

It is the foundation for the others because it defines:

shared constants
enums/types
helper functions
command opcodes

You want those fixed before the modules start depending on them.

After that, go straight to jtag_master.vhd.

That is the technical core of the whole project. If the JTAG engine is not solid, everything above it becomes hard to trust:

boundary scan
pin access
I2C-over-boundary-scan

all depend on it.

So the real first development target is:

get jtag_master working as a standalone engine

with support for:

reset
shift IR
shift DR
optional shift IR then DR
debug toggle TCK
set TMS
